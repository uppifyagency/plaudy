## Bugs & findings (priority order)

**BUG-1 — Dictation paste (and global shortcuts) silently dead when the frontend never initializes them. FIXED 2026-07-05 (same session), live-validated.**
True root cause (deeper than first hypothesis): Enigo + global shortcuts are initialized **only** by the frontend (`App.tsx:60-70`, gated on onboarding == `done`) — temporal coupling across processes. Any state where the webview doesn't run leaves paste/shortcuts silently dead: (a) dev-profile binary without vite (webview points at `localhost:1420` → dead, even with the window visible — this was the audit repro); (b) accessibility revoked (macOS drops it on signature change) while `start_hidden=true` → onboarding never seen.
Fix (double net, TDD, 2 new tests → 104 green): backend initializes Enigo+shortcuts at startup when accessibility is already granted (`lib.rs` setup, idempotent); `should_force_show_permissions_window` now covers macOS accessibility loss (was Windows-only). Live acceptance re-run on the exact repro: startup logs `Shortcuts initialized successfully`, dictation pastes, no `Failed to paste` error. Final check deferred to the Plaudy `.app`: fresh signature → force-show must appear once.

**FINDING-2 — `drop_bleed` misses when ASR transcribes the echo differently (<70 % word overlap).**
Known ceiling (transcript-level net; real fix = AEC, HANDOFF §11.2) — but now evidenced live: same-sentence echo survived as `Me`+`Speaker 1` duplicate. Levers: lower threshold / fuzzier match / AEC. Headphones or System mode avoid it entirely.

**FINDING-3 — Mic-only sessions are labeled `Speaker 1`, not `Me`.**
By design (`finalize_session` reserves "Me" for the meeting mic track), but the History card infers its source icon from labels → mic-only and system-only sessions are indistinguishable in the UI. Cheap fix: label single-mic sessions "Me", or persist the source on the row.

**FINDING-4 (dormant) — retention `PreserveLimit` would prune sessions to `history_limit=5`.**
All rows have `saved=0`; `cleanup_old_entries` is inert only because retention is `never` ([history.rs:749](../handy/src-tauri/src/managers/history.rs#L749)). If a user flips retention, meeting sessions get deleted like dictations. Fix idea: set `saved=1` on session rows, or exempt session rows from count pruning.

## Verdict

The product thesis is wired and working end-to-end (capture → diarize → merge → history → MCP). One real bug (BUG-1, hits fresh installs), one evidenced ceiling (FINDING-2), two design/product traps (3, 4). No crashes, no data loss, DB clean.
